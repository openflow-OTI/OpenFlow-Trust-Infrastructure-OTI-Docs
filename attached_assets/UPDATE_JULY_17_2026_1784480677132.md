# PART ONE — How OTI Was Originally Built

To understand what changed and why, you first need to understand the system that existed before any of the sessions documented here. This is that system.

## The Core Idea

OTI scores any blockchain wallet address from 0 to 100 using five weighted behavioral signals pulled entirely from public on-chain data. No human judgment, no whitelists, no KYC. The score is deterministic — the same wallet scored twice will return the same number (within the same data window). The five signals and their weights are:

- **Wallet Age** — 25%: how long the wallet has existed, measured from its first transaction
- **Transaction Count** — 20%: volume and frequency of on-chain activity
- **Token Holding Behavior** — 20%: diversity and quality of tokens the wallet holds or has interacted with
- **Smart Contract Interactions** — 20%: depth of engagement with DeFi protocols, NFT contracts, and other smart contracts
- **Transaction Timing Patterns** — 15%: whether the transaction pattern looks organic (healthy variance) or robotic (suspiciously regular)

These weights and the scoring logic inside `scoring.ts` are OTI's core protected IP. That file has never been touched and never will be — all fixes work around it.

The final score formula: `(walletAge × 0.25) + (txCount × 0.20) + (tokenHolding × 0.20) + (contracts × 0.20) + (timing × 0.15)`.

## How the Original Scoring Pipeline Worked

OTI was designed as a **reactive, pull-based system**. Nothing happened unless a user typed a wallet address into the frontend or made an API call. When that happened:

1. The frontend (React + Vite on Vercel) sent `GET /api/score/:address?chain=ethereum` to the backend
2. The backend (Express + TypeScript on Railway) validated the address format and chain
3. It checked an in-memory LRU cache (500 entries, 5-minute TTL) — if the score was cached from a recent lookup, it returned that immediately
4. On a cache miss, it called the appropriate external blockchain data API for the chain in question
5. That raw data was fed into `scoring.ts` which computed the five signals and the weighted final score
6. The result was written simultaneously to the in-memory LRU cache and the `chain_scores` database table
7. The score response was returned to the caller

The key phrase is "an in-memory LRU cache." Nothing survived a Railway restart. Every time the backend process died and restarted — which Railway does on deploys, on crashes, on memory pressure — the entire cache was empty. Every wallet that anyone had ever looked up was gone from the system's memory. The next person who asked about any of those wallets triggered a completely fresh set of external API calls.

## The Data Sources

For the 10 EVM chains (Ethereum, Polygon, Arbitrum, Avalanche, Sonic, Linea, zkSync, and the three intentionally disabled ones — BSC, Base, Optimism), data came from the Etherscan V2 API using a single API key: `ETHERSCAN_API_KEY`. The Etherscan V2 API allows routing requests to any supported chain via a `chainid` parameter, so one key served all 10 chains. Avalanche and zkSync had their own alternative endpoints (RouteScan and the zkSync block explorer API) but the same pattern.

For non-EVM chains:
- **TON** — Toncenter V2 API (with optional `TONCENTER_API_KEY` for higher rate limits)
- **Solana** — Solana mainnet JSON-RPC (public endpoint, no key)
- **Sui** — Sui fullnode RPC (public endpoint, no key)
- **Bitcoin** — mempool.space REST API (public, no key)
- **Tron** — TronScan API (`TRONSCAN_API_KEY`)

## The Database Tables

Six tables existed from the start, each serving a specific purpose:

**`chain_scores`** — the historical ledger. Every score ever computed, with full signal breakdown, wallet, chain, and timestamp. This was OTI's accumulating data moat — the more wallets were scored, the richer this table became. `persistScore()` wrote to it on every fresh score computation.

**`scores`** — a lightweight fast-lookup table. Just wallet + overall score, no signal detail. Designed for fast reads where you only need the number, not the breakdown. Written simultaneously with `chain_scores`.

**`wallets`** — the wallet address registry. Every address that has ever been scored gets a row here. Addresses plus chain family (evm, ton, solana, etc.). The linking table that both `chain_scores` and `wallet_links` reference by `wallets.id`.

**`wallet_links`** — future portfolio clustering infrastructure. Allows OTI to record that two wallet addresses belong to the same person. The schema existed from early in the project; no routes used it yet because it depends on WOR (users can't link wallets without first proving they own each one).

**`plan_configs`** — the rate limiting tier system. `anonymous` (3/day by default), `free`, `pro`, `enterprise`. The API key middleware reads `daily_limit` from this table dynamically on every request.

**`subscriptions`** — the API key table. Every external developer who integrates OTI gets a row here. Paired with `plan_configs`, these two tables form the complete monetization backend.

**`compromised_wallets`** — the trust denylist. Any wallet address here gets a `"compromised": true` flag in every score response, regardless of its calculated score. In the original design, only Ahmad (via admin endpoints) could add wallets here.

## The Admin System

The backend had a full admin API at `/api/admin/*`: create/edit/delete API keys, view usage stats, flush cache, add compromised wallets, view query history. This was the operational nerve center. The problem: **it had zero authentication**. Anyone who knew the URL could call any admin endpoint. There was no `x-admin-secret` header check, no IP allowlist, nothing. It was a fully open CRUD API for OTI's most sensitive operations.

## What the Business Model Looked Like

At this point in time, OTI's revenue strategy had three layers:

1. **API access** — developers pay for API keys with higher daily limits. Free anonymous tier (3/day) for casual use. Pro and Enterprise tiers for real integrations.
2. **Attestation** — a planned BAS (BNB Attestation Service) layer where OTI would issue blockchain-verifiable trust attestations. The plan was to charge per attestation.
3. **Widget distribution** — partners embed OTI's widget to show trust badges next to wallet addresses on their sites. The widget would query BAS attestation records and display the tier badge. Partners would pay a subscription for commercial widget use.

The planned distribution flow was: OTI bot replies in a Telegram/Discord group → community sees OTI → developers visit the site → get API key → integrate into their DeFi protocol → their users see wallet trust scores → some users visit OTI → some attest → some enterprise buyer sees OTI in use → pays for enterprise contract.

The widget was the passive distribution channel for this funnel. Partner sites would embed the badge, users would see it while connecting their wallets, and eventually some of those users would navigate to OTI to explore further.

# PART TWO — What Was Actually Wrong With the Original System

This section documents every significant problem discovered in the original system, what that problem caused, and how it was fixed. These are not minor polish items — several of them fundamentally changed what OTI was capable of.

## Problem 1 — The Pagination Trap: OTI Was Only Seeing a Fraction of Every Wallet's History

The single biggest class of data quality bug in the original system was pagination. Every external blockchain API (Etherscan, mempool.space, Toncenter, Solana RPC, Sui RPC, TronScan) paginates its results. The original fetcher code never paginated. It made one API call, took whatever came back, and treated that as the wallet's complete history.

**For EVM chains:** Etherscan caps results at 1,000 records per page for the transaction list endpoint and 500 per page for token transfers and internal transactions. A wallet with 5,000 native transactions received a score based on the most recent 1,000. Its oldest transaction — which determines wallet age — was invisible. A wallet that had been active since 2018 looked like it started in 2022, because 2022 was the start of the last 1,000 transactions.

For the Bitcoin chain, mempool.space's public API returns approximately 50 transactions per page without pagination. The Satoshi wallet — one of the oldest addresses on the Bitcoin blockchain, with thousands of transactions and a creation date of 2009 — was returning `walletAgedays: 5`. The code was reading the most recent 50 transactions and computing age from the oldest of those, which happened to be a few days ago. OTI was calling the Satoshi wallet a fresh wallet.

For TON, Solana, Sui, and Tron, similar single-page caps existed. Active wallets with years of history were being evaluated on a tiny slice of recent activity.

**What was fixed:** Each chain's fetcher now loops through pages until it either reaches the actual end of the wallet's history (a short page signals the last page), or until it hits a hard ceiling beyond which the API won't go. For EVM chains, that ceiling is 10,000 records (10 pages × 1,000 per page — Etherscan's real maximum). For Bitcoin, it paginates backwards through mempool.space's history endpoint until it finds the first transaction, with a 40-page safety cap. When OTI hits a cap before finding the true beginning, the response now discloses this via `historyTruncatedByApiCap: true` — an honest signal to API consumers that the wallet's full history may extend further than what's reflected.

## Problem 2 — Rate-Limit Errors Were Being Cached as Zero-Score Wallets

When Etherscan returned a NOTOK response (which happens during rate limiting, transient errors, or network issues), the original fetcher did something catastrophic: it treated NOTOK as "this wallet has no transactions" and returned an empty array. The scoring engine then computed a score for a wallet with zero activity — which scores low. That false low score was then written to the cache and the database.

The result: any wallet that happened to be scored during an Etherscan rate limit period was tagged as HIGH RISK in OTI's cache for the duration of that cache TTL, and written to `chain_scores` as a low-trust wallet. A busy DeFi user with years of activity could appear as a dangerous wallet because OTI happened to ask Etherscan for their data at the wrong millisecond.

**What was fixed:** Any NOTOK response that isn't specifically "No transactions found" (the legitimate empty-wallet case) now throws an error. An error in the scoring pipeline causes the request to return a 502, which is never cached. A false zero is never cached as a legitimate score. The system got one retry for transient failures before treating it as a real error.

## Problem 3 — Token Holdings Measured the Wrong Thing

The token holding signal was supposed to measure how many tokens a wallet currently holds and what quality those holdings represent — a proxy for whether the wallet belongs to an active crypto user who accumulates and holds assets versus a throwaway address.

The original implementation counted unique contract addresses seen in the wallet's transfer history. Every contract address that ever appeared in an incoming or outgoing token transfer was counted as a "held token." A wallet that received 20 tokens and immediately transferred all of them away showed the same token diversity score as a wallet that holds 20 tokens. Past interactions were indistinguishable from current holdings.

**What was fixed:** For EVM chains, OTI now queries Etherscan's `tokenbalance` endpoint for each token contract the wallet has interacted with (capped at 30 lookups to keep latency reasonable). Only contracts where the current on-chain balance is nonzero are counted as actual holdings. The scoring signal still comes from `scoring.ts` using transfer history (that file is sacred and untouched), but the API response now additionally includes `currentTokenHoldings` — an honest, real-balance view alongside it.

## Problem 4 — Token-Only Wallets Had Zero Age

Wallet age is computed from the timestamp of the wallet's first native transaction. Some wallets are pure receiving addresses — they receive ERC-20 tokens transferred by exchanges or other wallets on their behalf, but have never sent a native ETH transaction. For those wallets, the native transaction list (`txlist`) is empty. Empty `txlist` → `walletAge: 0` → scored as a brand-new wallet, regardless of how old the address actually is.

A wallet that had been receiving tokens for three years showed `walletAgedays: 0` because it had never paid gas itself.

**What was fixed:** When the native transaction list is empty, the backend now falls back to the earliest token transfer timestamp as the wallet age origin. The `earliestTokenActivityTimestamp()` function scans the token transfer history and returns the oldest timestamp. This is used to re-derive the wallet age score before the response is built — without touching `scoring.ts`. The overwrite happens in `score.ts`, which already has a pattern of post-hoc signal correction for chains where the standard EVM logic doesn't apply.

## Problem 5 — Non-EVM Chains Were Scored With EVM Signals

`scoring.ts` computes five signals for any wallet using an EVM-shaped data structure. The signals for Token Holding and Smart Contract Interactions are EVM-native concepts: tokens are ERC-20 contracts, contracts are Ethereum accounts with bytecode. Bitcoin has neither. Bitcoin addresses hold Bitcoin. Bitcoin doesn't have ERC-20 tokens. Bitcoin doesn't have smart contracts. Bitcoin wallets were being scored on Token Holding (always 0) and Smart Contract Interactions (always 0), and those zeros were being multiplied by their 20% weights each, dragging Bitcoin wallet scores down by up to 40 points for signals that simply don't apply.

The Satoshi wallet was being called a medium-trust wallet because it had zero token diversity and zero contract interactions — not because Satoshi is untrustworthy, but because Bitcoin doesn't have those things.

**What was fixed:** `chainWeighting.ts` was added (a new file — `scoring.ts` is still untouched). For Bitcoin specifically, the weight budget allocated to Token Holding (20%) and Smart Contract Interactions (20%) — the two non-applicable signals — is redistributed proportionally across the three signals that do apply (Wallet Age, Transaction Count, Timing Patterns). Bitcoin wallets now score based entirely on signals that mean something for Bitcoin. The raw signal values that `scoring.ts` produces are still returned honestly in the response (Token Holding = 0, Contracts = 0 for Bitcoin) — but the final score is computed using the redistributed weights.

## Problem 6 — Smart Contract Diversity Was Measuring the Wrong Address (The to=self Bug)

For every non-EVM chain (TON, Solana, Tron, Sui), the original code that computed "unique smart contracts interacted with" was using the wallet's own address as the destination of every transaction. It was asking "how many distinct contracts did this wallet call?" but using the wallet itself as the answer to "where did these transactions go?"

The practical result: every wallet had a unique-contract-interaction score of 1, regardless of how many real contracts they'd used. A wallet that had interacted with 50 different Solana programs over three years showed the same contract diversity as a freshly created wallet with one transaction.

**What was fixed:** Each non-EVM chain now extracts the real destination address from transactions. For Solana, a sampling step parses up to 300 transaction details and extracts real program IDs. For TON, outgoing message destinations are used instead of the self-referential field. For Tron, only TriggerSmartContract transaction types are counted (bandwidth/energy delegation, freeze, and vote transactions were being included, inflating the count with internal-housekeeping calls that have nothing to do with DeFi engagement). For Sui, real object addresses are extracted.

## Problem 7 — Receive-Only Wallets on TON and Sui Showed 0-Day Age

The same class of bug as Problem 4, but affecting TON and Sui. TON's fetcher was filtering transaction history to only outgoing messages. A wallet that had only ever received TON (common for wallets set up to receive but not yet spend) had zero outgoing messages and therefore `walletAgedays: 0`. Sui's transaction query filtered by `FromAddress` only — a receive-only Sui wallet appeared to have no transaction history.

**What was fixed:** TON now examines both inbound and outbound history and uses the earliest timestamp across both. Sui queries both the `FromAddress` and `ToAddress` directions and takes the earliest timestamp across both result sets.

## Problem 8 — The DB Cache Was Write-Only (Data Was Accumulating and Being Ignored)

The `chain_scores` table was written to after every fresh score computation. The `wallets` table was being populated. The database was accumulating real, valuable data on every wallet ever scored. But the scoring endpoint never checked this data before making external API calls.

The flow was: request arrives → check in-memory LRU → if miss, call Etherscan/Toncenter/etc. → compute score → write to DB. There was no step between "LRU miss" and "call external API" where the DB was consulted. All that data sitting in `chain_scores` was completely ignored during score requests. It was written and forgotten.

This meant:
1. Every Railway restart sent 100% of traffic back to external APIs, at full latency
2. Wallets that were scored months ago and hadn't changed were re-fetched from Etherscan every time the cache expired
3. OTI was spending its rate limit budget on wallets it already knew everything about

**What was fixed:** A two-tier cache architecture (BF13). L1 is the in-memory LRU — unchanged, same 5-minute TTL, same 500-entry cap. L2 is the `chain_scores` database table. Every L1 miss now checks L2 before touching any external API. If a score exists in `chain_scores` that was computed within the configured rescore window (default 30 days), L2 returns it immediately, warms L1, and the request completes with zero external API calls.

The rescore window is configurable. A 30-day window means a wallet that was scored in the last month won't trigger any Etherscan API calls — OTI already knows this wallet well enough. A wallet scored more than 30 days ago gets a fresh compute (it may have changed). Ahmad can adjust this window from the admin panel without touching any code. The setting lives in `system_settings`, read with a 60-second in-memory TTL so admin panel changes propagate within one minute.

**Keep-highest logic on writes:** When a fresh score is computed and written to `chain_scores`, if a higher score for that wallet already exists in the table, the higher score is preserved — only the `scored_at` timestamp is updated to reset the rescore window. A wallet's score can only go up permanently, not down. This is fair: a wallet that had a good history and then had a period of inactivity should not be penalized by a lower score computed during that quiet period.

**Cache toggle:** The entire cache can be turned off from the admin panel. When off, every request fetches live chain data, no DB reads or writes for scoring. This is a research/debugging tool — not for normal operation.

## Problem 9 — The Dead Memory Write Was Still Running After the DB Was Fixed

Once the history endpoint was fixed to read `chain_scores` from the database (instead of the in-memory Map in `lib/history.ts`), the old write-to-memory code was still running inside the scoring pipeline on every fresh compute. Every score was being written to a Map that nothing read from anymore. Memory being burned on every request for a data store with zero readers.

`lib/history.ts` was deleted. The `recordHistory()` call was removed from `score.ts`. The grep confirmed nothing else imported it.

## Problem 10 — Chain Routing Logic Was Scattered Across Four Files

The list of supported chains, the chain-to-family mapping, and the set of chains that require Etherscan plan upgrades (BSC, Base, Optimism) were defined inline in `score.ts`, `history.ts`, and `chainFamily.ts` in three separate places. Adding a new chain meant finding all four locations and updating each one — easy to miss one, easy to get them out of sync.

**What was fixed:** `chainRegistry.ts` is now the single source of truth. It exports `SUPPORTED_CHAINS`, `ALL_CHAIN_ZOD_ENUM`, `CHAIN_FAMILY_MAP`, and `PLAN_UPGRADE_REQUIRED`. Every route that needs to know about chains imports from there. Adding a 16th chain means editing one file.

## Problem 11 — Railway Never Ran Database Migrations Automatically

When a Builder added a new database table or column, they'd write a Drizzle migration, commit it, and assume Railway would apply it on the next deploy. Railway didn't. Railway's build process ran `pnpm install`, ran the build, and started the server. It never ran `drizzle-kit push`. Every schema change required Ahmad to manually open the Railway console, navigate to the database panel, and run the migration command by hand.

This was discovered painfully when multiple features shipped with database dependencies that were silently broken on production until Ahmad happened to notice. The `system_settings` table didn't exist on the Railway production DB when BF13 shipped, so every settings read fell back to defaults and every admin panel settings change silently failed.

**What was fixed:** `railway.json` now runs `pnpm --filter @workspace/db run push` as part of the build command, after the api-server build, before startup. Plain `push` was chosen deliberately over `push-force` — additive changes (new tables, new columns) apply automatically, but ambiguous or destructive changes still require a manual decision instead of silently being skipped. `nixpacks.toml` was not touched.

## Problem 12 — Admin Endpoints Had Zero Authentication

Any person on the internet who knew that `https://workspaceapi-server-production-5c0c.up.railway.app/api/admin/keys` existed could create API keys, delete API keys, change plan limits, or flag arbitrary wallets as compromised. This wasn't a theoretical risk — the backend URL is not a secret (it's in the frontend's Vercel environment variables and appears in network requests from any browser), and the admin API is fully documented in the OpenAPI spec.

**What was fixed:** `adminAuth.ts` middleware was added, applied to the entire `/api/admin/*` router. Every admin request must include an `x-admin-secret` header matching the `ADMIN_SECRET` environment variable set in Railway. Wrong or missing secret returns a generic 401. `ADMIN_SECRET` is an environment variable — never committed to the repo, never in any file.

## Problem 13 — Fantom Was Renamed Sonic (and Ghost Chains Were Listed)

Etherscan V2 chain ID 146 was labeled "Fantom" in OTI's codebase and documentation, but chain ID 146 is actually Sonic Mainnet — Fantom Opera migrated to Sonic. OTI was querying Sonic data and calling it Fantom. Additionally, Scroll, Sepolia, and Holesky appeared in some frontend and documentation references as "supported chains" but were not implemented in the backend at all.

**What was fixed:** The `fantom` → `sonic` rename was applied across `etherscan.ts`, `chainRegistry.ts`, `chainFamily.ts`, and every frontend/docs reference. Scroll, Sepolia, and Holesky were confirmed absent from all backend code (they were already not implemented — the docs just mentioned them incorrectly).

# PART THREE — The Wallet Ownership Registry (WOR)

## What Existed Before WOR

Before Task 16, compromise reporting worked like this: a wallet owner whose private key was stolen would need to contact Ahmad. Ahmad would manually add the wallet address to the `compromised_wallets` table via the admin API. Until Ahmad did this, the stolen wallet continued to show its trust score as if nothing was wrong. The same trusted address that an attacker now controlled would show as HIGHLY TRUSTED in every OTI query — the opposite of what users needed to see.

The `wallet_ownership` and `wallet_links` tables existed in the database schema from early in the project as infrastructure for a future feature. Zero routes connected to them. They were placeholders.

## The WOR Design and Why the Passkey Is the Differentiator

WOR is built on a specific threat model: an attacker who has stolen a private key controls the wallet. They can sign any transaction the original owner could sign. They can prove "ownership" to any system that relies on wallet signatures alone. Standard wallet authentication — "sign this message with your wallet" — provides no protection against a stolen key because the attacker has the key.

The passkey breaks this. Before any compromise happens, the wallet owner registers with OTI in two steps: they sign a challenge message with their wallet (EIP-191 signature), and they create a secret passkey (minimum 8 characters, stored as a bcrypt hash at cost 12). The passkey is stored nowhere the attacker can find — it lives only in OTI's database as a hash, and in the wallet owner's memory.

When the wallet is later compromised, the real owner reports it by providing both factors: a fresh wallet signature and the passkey. Both must verify. If the attacker has the key, they can provide the signature but not the passkey. If someone guesses the passkey but doesn't have the key, they can't provide the signature. Neither factor alone is enough.

Critically, both checks always fail with the same generic 401 error — "Verification failed" — regardless of which factor was wrong. This means an attacker who has the key cannot enumerate passkeys by watching whether the response changes — both "wrong passkey" and "wrong signature" look identical from the outside.

When both factors verify, the compromise report is atomic: `wallet_ownership.status` is set to `"compromised"`, the address is inserted into `compromised_wallets` with the reason "Self-reported by owner via WOR," and all cached scores for that address are invalidated immediately. From that moment forward, every OTI query for that wallet returns the COMPROMISED flag — regardless of the wallet's calculated trust score.

## The Challenge Replay Protection

Each registration and compromise report uses a fresh time-embedded challenge message. The challenge includes a timestamp, and challenges expire after 15 minutes. Even if an attacker intercepts a challenge-signature pair, they cannot reuse it — it will have expired. The `extractTimestampRaw()` function parses the timestamp from the message, and `isChallengeExpired()` rejects any challenge older than 15 minutes. The message must also match an exact expected template — crafted messages designed to trick the signing step are rejected by `messageMatchesTemplate()`.

## The Admin Endpoints

Four admin endpoints were added alongside the WOR user-facing endpoints: paginated view of all registered wallets, paginated view of compromised wallets, manual admin flag, and manual admin unflag. The compromised view sources from `compromised_wallets` table (the true source of truth, the same table the public score endpoint reads from), with a LEFT JOIN to `wallet_ownership` for additional metadata when available. This matters because an admin can manually add a wallet to `compromised_wallets` without that wallet having ever registered for WOR — the admin view must show both cases.

## Why wallet_links Is Still Infrastructure Only

`wallet_links` allows two wallet addresses to be linked as belonging to the same person. The routes were implemented as part of Task 16. The reason it has no UI and no business function yet: linking requires both wallets to be registered with WOR first. Without WOR having significant adoption, wallet linking has no population to work with. The infrastructure is ready — once WOR is live and users are registering, the portfolio view and cross-wallet trust aggregation that `wallet_links` enables can be built on top of it without any schema changes.

# PART FOUR — The Strategic Pivot: From Reactive to Proactive

This is the most significant directional change in OTI's history. Every session before July 17, 2026 was building a reactive system. July 17 was when we designed a proactive one.

## The Original Design Was Pull-Based

In the original design, OTI scored wallets when people asked. A user on a DeFi protocol connects their wallet → that action triggers an API call (either from the partner's backend or from the embedded widget) → OTI fetches blockchain data, computes the score, returns it. The wallet didn't exist in OTI's system until someone asked about it.

This created a cold-start problem for every distribution channel. The bots couldn't reply with a score unless someone typed a command. The widget couldn't show a badge unless the connecting wallet had already been scored by someone at some point. An exchange couldn't use the API to screen wallets unless OTI had data on those wallets. OTI's usefulness to any distribution channel was directly proportional to how many wallets OTI had already been asked about — and OTI had only been asked about the wallets of people who had already found OTI.

## The New Design Is Push-Based: Fetching Directly From On-Chain

The direction we chose on July 17: OTI does not wait for people to ask. OTI fetches wallet data directly from the blockchain and scores wallets in background batches, proactively, before anyone asks.

The mechanism: blockchain explorers and public RPC endpoints make transaction data available for any address. OTI's backend runs a background scoring pipeline that identifies wallet addresses from public on-chain data — addresses that appear in recent transactions, active DeFi protocol participants, newly active wallets — and scores them without any wallet owner or API consumer having to request it. By the time a user connects their wallet to a partner site, their badge is already waiting in OTI's database.

This changes OTI from a lookup service ("ask us about a wallet") to an ambient intelligence layer ("we already know about your wallet"). The cold-start problem is solved at the infrastructure level rather than the distribution level.

The practical implication for the fetcher layer: this significantly increases the volume of Etherscan API calls. In the reactive model, Etherscan was called only when a human or bot triggered a score request. In the proactive model, OTI is continuously making Etherscan calls as part of background batch scoring — calls that are not triggered by any user interaction. The rate limit budget is being consumed by the infrastructure itself.

## What Changes Downstream of This Decision

Everything in the original distribution strategy assumed wallets needed to be scored on-demand. The widget existed to trigger score requests at the point of a wallet connection. The bots existed to trigger score requests via commands. Once OTI is proactively pre-scoring from on-chain data, those distribution channels change purpose: they stop being scoring triggers and become discovery surfaces.

A user who connects their wallet to a DeFi protocol with OTI's widget embedded doesn't trigger a score request — the widget looks up a score that already exists. The widget becomes ambient context rather than an on-demand lookup. A Telegram bot reply that includes an OTI badge is delivering pre-computed intelligence rather than triggering a computation. The distinction matters architecturally: latency drops to a database lookup, and reliability stops depending on Etherscan being available at the moment of the query.

# PART FIVE — The Score Source Architecture

## The Problem With a Hardcoded Data Source

The original widget design connected directly to BAS (BNB Attestation Service): the widget queried BAS for an attestation record, found the tier, and displayed the badge. This was the only planned data source. The widget code was designed to work with BAS and only BAS.

The problem: attestation requires wallet owners to have already found OTI, visited the site, gone through the registration flow, and paid for or received a free attestation. During any early growth period, most wallet addresses have no BAS attestation. The widget renders nothing for those wallets. A partner site that embeds OTI's widget during the first months of distribution shows empty badges next to most wallet connections — which is indistinguishable from the widget not working at all.

Changing the data source later would require updating the widget code, redeploying it, notifying all partners, and asking them to update their embed code — a coordinated migration event that gets harder with every new integration.

## The Score Source Switcher

The solution is to make the data source a server-side configuration setting rather than a hardcoded client-side decision. The admin panel will expose a **Score Source** selector with three options:

**OTI Backend** — the widget API queries OTI's own database (the `chain_scores` table populated by proactive background scoring). No blockchain dependency. No BAS dependency. No attestation required. Available from day one. This is the correct default during early growth.

**BAS Attestation** — the widget API reads attestation records directly from BNB Chain via the BAS SDK. Trustless, blockchain-verifiable. Only returns data for wallets that have an active BAS attestation. Appropriate once attestation adoption reaches meaningful scale.

**Auto (fallback)** — tries BAS first. If a BAS attestation exists for this wallet, returns it. If not, falls back to OTI's database score. This is the intended long-term default: trustless verification when available, always-available fallback when not.

This setting lives in the `system_settings` table (already exists). The widget API endpoint reads it on every request via the `getSettings()` function (60-second in-memory TTL — already implemented). When Ahmad changes the setting in the admin panel, every embedded widget worldwide updates within one minute. No partner deploys, no partner notifications, no embed code changes.

The widget's embed code is agnostic to this decision by design. Partners set a `data-wallet` attribute, a `data-chain` attribute, and their `data-key`. They never specify where the data comes from. That routing happens entirely on OTI's servers, invisibly to the partner.

# PART SIX — The Business Model: What Was Missing

## Partners Had No Financial Reason to Embed the Widget

The original business model positioned the widget as a trust UX enhancement for partner sites: "your users see wallet trust badges — this improves their experience." The partner's incentive was purely UX. OTI would eventually charge partners a subscription fee for commercial widget use. From the partner's perspective, the widget was a cost center with a UX benefit.

This is a weak incentive structure for a B2B distribution channel. The strongest incentive for any partner to embed a third-party tool is when that tool generates measurable revenue. Analytics scripts, ad tags, affiliate links, payment processors — these all get priority treatment from engineering teams because they generate revenue. A trust badge that costs money generates no revenue.

## Partner Revenue Share

Partners earn a percentage of every attestation fee generated through their embedded widget. When a user discovers their OTI badge on a partner site and claims the full cryptographic BAS attestation through the in-widget flow, a share of that attestation fee is credited to the partner.

The percentage is Ahmad's decision to configure via the admin panel — it is not hardcoded. What is locked is the model: **partner-attributed attestation = partner receives a share.**

The infrastructure implication: every attestation payment record must carry attribution metadata from the moment the Phase 3 billing infrastructure is built. An attestation triggered through widget embed A on partner site P must know its origin. This is not optional and cannot be retrofitted — attribution tracking must be built into the payment record schema on day one of Phase 3, or the revenue share model has no data to operate on.

This alignment changes the partner relationship permanently. A high-traffic DeFi protocol embedded with OTI is now earning a passive revenue stream from its user base's attestations. Their incentive is to promote the attestation flow, surface the widget prominently, and design UX that encourages users to claim their badge. OTI doesn't need to manage this — the incentive structure manages it.

## The Token Airdrop Was Too Narrow

DECISIONS.md D22 and TOKENOMICS.md Section 7.2 defined the early-adopter airdrop as: "the first 1 million wallets that register for OTI attestation receive OTI tokens." The airdrop was a loyalty reward for existing OTI users — people who had already found the product and completed the attestation flow.

The strategic purpose of an early-adopter airdrop is discovery, not loyalty. A wallet owner who receives unfamiliar tokens in their wallet investigates. They find "OTI." They discover they already have a trust score. That discovery event is passive acquisition — no marketing spend, no outbound campaign, no user action required.

But if every airdrop recipient has already completed attestation, they've already discovered OTI. The discovery moment is lost. The airdrop reaches people who didn't need to be found.

**The change:** Airdrop eligibility is expanded to the first 1 million wallet addresses in OTI's scoring database — including wallets that OTI has proactively pre-scored, regardless of whether the wallet owner has ever visited OTI or knows OTI exists. When the token launches, OTI sends tokens to those addresses. The owners check their wallets, see unfamiliar tokens, search "OTI," and discover they already have a trust score. The discovery happens as a consequence of the airdrop rather than being a prerequisite for it.

The tracking implication: the list of first-1M eligible wallet addresses is recorded from the moment proactive background scoring begins — not from the moment attestation launches. Every wallet OTI proactively scores is a potential airdrop recipient.

## API Tier Signal Depth: The Boundary That Protects Revenue

The current API response includes `{ score, weighted, maxWeight }` per signal — the full breakdown. This is Pro tier information. But the free tier and the BAS attestation tier should return only the tier label and score. If free-tier API consumers get the full signal breakdown, there is no compelling reason to upgrade to Pro.

The confirmed tier boundaries:

**Free / Anonymous:** Score number + tier label + compromised flag. This is what the BAS attestation also stores. Free tier matches attestation — neither competes with the other, they serve the same information through different channels (live API vs. blockchain record).

**Pro:** All five signal scores with weighted contributions (`score`, `weighted`, `maxWeight` per signal) + score history (the last N scores for that wallet from `chain_scores`). This is the intelligence layer — the data that explains why a wallet received its tier. BAS attestation stores only the tier, not the breakdown. Any developer who needs signal-level intelligence (to risk-price a loan, to adjust collateral requirements, to audit a wallet's history) must use the Pro API. The attestation creates the public proof; the Pro API unlocks the private intelligence.

**Enterprise:** Everything in Pro + raw per-chain signal numbers (each individual chain the wallet is active on, not just the unified cross-chain EVM score) + webhook alerts for watched-wallet score changes + batch scoring endpoint + compliance-formatted export. This is the data infrastructure layer for exchanges, payment processors, and regulated entities who need OTI's scoring in their operational pipelines.

The key structural design decision: **BAS attestation stores tier only, never signal breakdown.** This is not a limitation — it is deliberate. It ensures the attestation and the Pro API are permanently complementary. Changing this would destroy the API revenue model.

# PART SEVEN — The Badge and Widget Design

## The Badge Visual Design (Finalized July 17, 2026)

**Score thresholds and tier identities:**

| Tier | Label | Score Band | Colour | Hex |
|---|---|---|---|---|
| 1 | HIGHLY TRUSTED | 75–100 | Mint green | `#00E5A0` |
| 2 | TRUSTED | 55–74 | Sky blue | `#4FC3F7` |
| 3 | NEUTRAL | 35–54 | Cool gray | `#90A4AE` |
| 4 | RISKY | 20–34 | Amber | `#FFB300` |
| 5 | HIGH RISK | 0–19 | Red | `#FF4444` |

The score thresholds in the existing ROADMAP.md used different bands (85/65/45/25). Those are replaced by the finalized bands above.

**Compromised override:** Any wallet in the `compromised_wallets` table gets the Compromised badge regardless of its trust score. A HIGHLY TRUSTED wallet that self-reports a compromise immediately shows the Compromised badge. Score is irrelevant once a wallet is flagged. Colour: same red as HIGH RISK (`#FF4444`) with a warning icon (!) replacing the checkmark.

**Visual anatomy (full card format — used on OTI's own pages):**
- Dark background, tinted to the tier colour, with a subtle ambient glow matching the tier
- Shield icon: pointed outline with checkmark inside. Fill: tier colour at 14% opacity. Stroke: tier colour at full opacity. Size: 44×44px in card form
- Score number: 28pt, tier colour, bold, below the shield
- Tier label: small caps, tier colour, below score number
- Horizontal divider: thin line at tier colour, 18% opacity
- "OTI Verified" footer: very small, subdued gray
- Chain logo bubble: 18×18px circle at bottom-right of the shield. Background: page background. Border: chain's brand colour. Text: chain ticker at chain's brand colour

**Chain bubble colours:**
- ETH: `#627EEA` | BNB: `#F0B90B` | POL: `#8247E5` | SOL: `#9945FF` | TRX: `#EF0027`
- All other EVM chains: their official brand colour, or neutral gray if none established

**Why this design language:** The shield communicates protection immediately — the same visual shorthand used by security software, government IDs, and trust seals globally. The colour coding follows the universal risk scale (green = safe, amber = caution, red = danger) that users already know from traffic lights and alert systems. OTI doesn't teach a new colour system — it uses the one that already exists in every user's head.

## The Badge Interaction Model (Finalized July 17, 2026)

The badge has three states. This is the confirmed interaction design for the widget's inline pill — the compact format that sits next to a wallet address on a partner's page.

**State 1 — Collapsed (always visible, no interaction required)**

A small horizontal pill sitting next to the wallet address. Contains: shield icon (14px) with chain bubble at bottom-right + score number + tier label + ▼ arrow. Background: tier colour at 12% opacity. Border: tier colour at 33% opacity. This is the ambient trust signal — always present, requires nothing from the user.

**State 2 — Hover (quick summary, desktop only)**

What triggers it: mouse-over on desktop (touch skips to State 3 on tap). What appears: a floating tooltip panel above the pill, pointing down. Contains: tier label and score number in tier colour (large) + all five signal bars (name left, gradient bar centre, score right) + wallet metadata chips (on-chain since date, transaction count) + "Click for full breakdown →" prompt at bottom. The tooltip disappears on mouse-out. It is a teaser — enough to understand the score at a glance without committing to the full breakdown.

**State 3 — Expanded (full inline breakdown panel)**

What triggers it: clicking the pill (or tapping on touch). What happens: the row expands in place. A panel appears directly below the wallet row in the partner's table or list. Nothing navigates away. The partner's page does not scroll. Contains: shield + tier + chain + metadata (on-chain since / N transactions) + score number large and right-aligned + all five signal bars with gradient fills + signal weight footnote + "OTI Verified ✓" tag + "Powered by OTI · otiscore.com" attribution. Clicking the pill again collapses the panel (arrow toggles ▼/▲).

**Compromised expanded state:** Red warning shield with "!" → "⚠ Wallet Compromised" heading → explanation: "This wallet was flagged by its owner via OTI's Wallet Ownership Registry (WOR). Private key or passphrase may be exposed. Do not send funds to this address." → report date of the flag.

**Why three states:** Ambient trust (State 1) serves the majority of users who just need a signal. Quick context (State 2) serves the curious who want to verify without disrupting their flow. Full transparency (State 3) serves anyone who needs to understand or challenge the score. A badge that only shows a tier label gives no recourse. A badge that immediately shows all five signals overwhelms the 95% of users who just need to see "TRUSTED."

The "no redirect" rule is absolute: every interaction — from hover tooltip to signal breakdown to WOR compromise reporting — must happen inside the widget panel on the partner's page. The moment the widget navigates a user away from the partner's site, the partner's flow is broken. Partners cannot tolerate a trust widget that ejects their users.

## WOR Inside the Widget

The WOR registration and compromise report flow are embedded as in-widget panels. A wallet owner who sees their badge on a partner site can initiate the full WOR flow (wallet signature + passkey + submit) without leaving the partner site. This applies to both first-time registration (pre-emptively establishing ownership before any compromise) and active compromise reporting.

This is not a convenience feature — it is a distribution decision. Most users who encounter OTI will encounter it on a partner site, not at otiscore.com. If WOR is only accessible at OTI's own URL, the safety flow is invisible to the majority of users most of the time. Making WOR accessible inside the widget means every partner becomes a WOR registration point.

The implementation uses OTI's existing WOR endpoints (`POST /api/wallet/register`, `POST /api/report/compromised`). CORS is already fully open — these endpoints can be called from any origin. The only new work is the widget's frontend WOR UI panels.

## The Widget Embed Specification

The widget is a single self-contained vanilla JS script tag:

```html
<script
  src="https://widget.otiscore.com/oti.js"
  data-wallet="0x..."
  data-chain="ETH"
  data-key="partner-api-key"
></script>
```

The script reads its own attributes, calls OTI's widget API endpoint, and renders the badge pill in the DOM at the exact position of the script tag. The widget API applies the Score Source setting server-side and routes the request accordingly (Backend DB / BAS / Auto).

**Four hard constraints that must be enforced at implementation time:**

1. **Zero external dependencies.** The script must be completely self-contained. No React, no Vue, no lodash, no fetch polyfill assumptions. Vanilla JS and the Fetch API only. Partners cannot be expected to have any specific technology stack. The embed must work on a PHP page from 2015 and on a Next.js 15 app equally.

2. **Scoped styles.** All CSS injected by the script must be scoped to its own elements — via a unique class prefix or Shadow DOM. It must not leak into the partner's page or be overrideable by the partner's global styles. CSS collision is the most common failure mode for embedded widgets.

3. **Graceful empty state.** If no score exists for the wallet (not in DB, not in BAS), the widget renders nothing. Silently. No error message, no broken placeholder, no loading spinner that never resolves. Absence is the correct behavior for wallets OTI has no data on yet.

4. **No redirect on any interaction.** Every user action — hover, click, WOR flow, attestation claim — happens inside the widget. The script never calls `window.location`, never opens a new tab, never navigates the user away from the partner's page.

## Badge Chain Display Logic

**Default (free tier):** The badge shows OTI's unified cross-chain EVM score — the wallet's aggregate score across all EVM chains OTI has data on. The chain bubble shows the chain the partner site is operating on (passed via `data-chain` attribute), or the chain with the most significant activity if none specified.

**Per-chain breakdown (Pro add-on):** Showing individual scores per chain in the expanded breakdown panel ("ETH: 88, Polygon: 71, Arbitrum: 65") is a Pro tier feature. Free tier shows the unified score only.

**Non-EVM chains (Solana, TON, Tron) — deferred:** Non-EVM address formats are completely different from EVM addresses. A Solana address is not the same address as the Ethereum address belonging to the same person. Linking them requires `wallet_links`, which requires WOR to exist and users to prove they own both addresses. Until WOR has adoption and `wallet_links` is populated, non-EVM chains cannot appear in the same badge view as EVM activity. OTI continues scoring non-EVM chains on their own, but the widget shows EVM unified scores only until the cross-chain identity infrastructure is in place.

# PART EIGHT — API Key Rotation and the Cost Problem

## What the Single-Key Architecture Means at Scale

OTI currently uses one `ETHERSCAN_API_KEY` for all EVM scoring. Etherscan's free tier has a daily request limit and a per-second rate limit (3 requests per second, which is why the throttle in `etherscan.ts` fires every 350ms — safely under the ceiling). Within those limits, data is free. Above them, Etherscan charges — and the jump to a paid plan is a real monthly cost.

In the reactive model (score only when someone asks), this was manageable. Real users trigger real requests at human speeds. The rate limiter and the LRU/DB cache absorbed most of the repeat traffic.

In the proactive model (background batch scoring of wallets from on-chain data), OTI itself is generating API calls that are not triggered by any user request. The scoring pipeline is continuously consuming Etherscan capacity as an infrastructure cost. The volume scales with how aggressively OTI chooses to pre-score. During early growth, before revenue can cover infrastructure costs, this creates a cost exposure cliff: below the free tier limit, everything is free; above it, every batch scoring call costs money.

## The Key Rotation Decision

The solution: a pool of free-tier Etherscan API keys, rotated round-robin across requests inside the fetcher layer. Each free key has its own daily request limit. Pooling N keys multiplies OTI's free-tier capacity by N before any paid API cost is incurred.

The rotation logic lives entirely inside `etherscan.ts` — the `etherscanApiKey()` function (which currently returns the single key from `process.env.ETHERSCAN_API_KEY`) would become a round-robin selector across an array of keys. No other file changes. The scoring logic is untouched. The cache logic is untouched. The admin panel is untouched. The API response shape is untouched. The rotation is invisible to everything upstream of the fetcher.

**Why this is a future task, not an immediate one:** Key rotation matters when proactive background scoring volume is high enough to threaten free-tier limits. That threshold depends on how many wallets are being pre-scored per day and how long it takes each one. Building the rotation now, before there is any background scoring infrastructure, is premature — the batch scorer and the fetcher pool are related but separate concerns. The rotation belongs in Phase 4 (Growth Features) because it is a cost management tool that becomes relevant when the background scoring pipeline is generating significant volume.

**Why key rotation rather than just paying for Etherscan:** During the period between early growth and first meaningful revenue, key rotation is the buffer that keeps infrastructure costs at zero. Once API subscription revenue and attestation fees are covering costs, upgrading to a paid Etherscan plan is the right path — it gives higher rate limits, more chains, and better SLAs. Key rotation is the bridge to that point, not the permanent solution.
